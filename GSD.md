# Get Shit Done (GSD) CLI 指令使用手册

本手册包含 GSD 框架中所有核心指令的翻译、功能描述及典型使用场景，旨在帮助开发者快速掌握该 AI Agent 工作流。

## 关联仓库 (Related Repositories)

- **核心框架**: [gsd-build/get-shit-done](https://github.com/gsd-build/get-shit-done) — GSD 自动化工作流的官方实现。
- **底层规约**: [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) — 提供标准化的 AI 代理工程域技能支持。

## /gsd-add-tests

- **功能简述**: 为完成的阶段生成测试
- **详细说明**: 根据 UAT 标准和具体实现，为已完成的阶段补充测试。
- **使用场景**: 业务代码已经写完并能跑通，但需要提高测试覆盖率，补充单元测试或集成测试时使用。
- **e.g.**: `/gsd-add-tests 为 auth.service.ts 生成基于 UAT 标准的 Jest 单元测试。`

## /gsd-audit-fix

- **功能简述**: 自主审计与修复流水线
- **详细说明**: 自动发现、分类、修复、测试并提交代码问题。
- **使用场景**: 希望 AI 自主全自动扫码代码库寻找漏洞，并直接提交修复代码时使用。
- **e.g.**: 周末运行 `/gsd-audit-fix`，让系统自动扫描并修复潜在的 SQL 注入风险。

## /gsd-audit-uat

- **功能简述**: 跨阶段审计 UAT
- **详细说明**: 对所有未结清的 UAT 及验证项进行跨阶段审计。
- **使用场景**: 需要对整个项目的用户验收测试 (UAT) 进行全面回顾，确保所有功能符合最初的业务验收标准时使用。
- **e.g.**: 发布候选版 (RC) 前，运行 `/gsd-audit-uat` 核对是否有漏掉的用户故事验证项。

## /gsd-autonomous

- **功能简述**: 自主运行剩余阶段
- **详细说明**: 按阶段执行“讨论-计划-执行”循环，无需人工干预。
- **使用场景**: 计划已经排满，你准备去喝咖啡，希望 AI 按照既定路线图自主完成后续所有编码和测试时使用。
- **e.g.**: 确认路线图无误后输入 `/gsd-autonomous`，系统将自动循环执行：讨论 -> 计划 -> 编码。

## /gsd-capture

- **功能简述**: 捕捉想法、任务与种子
- **详细说明**: 将灵感、临时任务或笔记快速记录到指定位置。
- **使用场景**: 在处理主线任务时突然来了灵感，或发现了不在当前范围的次要任务，需要快速记录以免遗忘时使用。
- **e.g.**: `/gsd-capture 记录：稍后需要优化首页的图片加载速度。`

## /gsd-cleanup

- **功能简述**: 归档已完成阶段的目录
- **详细说明**: 清理并归档已达成里程碑的相关阶段目录。
- **使用场景**: 当一个大版本（里程碑）发布完毕后，需要清理并归档开发过程中产生的临时阶段文件和目录时使用。
- **e.g.**: v1.0 版本上线后，运行 `/gsd-cleanup` 将相关的 `.planning` 文件打包归档。

## /gsd-code-review

- **功能简述**: 审查阶段变更文件
- **详细说明**: 检查源代码中的 Bug、安全性问题和代码质量缺陷。
- **使用场景**: 在阶段任务执行完毕后，要求 AI 扮演资深开发者的角色，对修改的文件进行深度代码审查时使用。
- **e.g.**: 运行 `/gsd-code-review` 检查刚刚重构的代码是否存在 O(n^2) 的性能隐患。

## /gsd-config

- **功能简述**: 配置 GSD 设置
- **详细说明**: 调整工作流开关、高级参数、集成选项和模型配置。
- **使用场景**: 需要调整 GSD 框架自身的高级设置，如修改底层大模型引擎、调整工作流开关等。
- **e.g.**: 运行 `/gsd-config` 切换 AI 引擎为 Gemini Pro 或调整并发执行的线程数。

## /gsd-debug

- **功能简述**: 系统性调试
- **详细说明**: 在上下文重置期间保持持久状态，进行系统化的错误排查。
- **使用场景**: 遇到复杂、顽固的系统错误，需要跨越多个上下文会话进行持久化状态跟踪排查时使用。
- **e.g.**: `/gsd-debug 排查生产环境下偶发的内存泄漏问题`，让 AI 代理持久化保留调试线索。

## /gsd-docs-update

- **功能简述**: 生成或更新项目文档
- **详细说明**: 根据现有代码库验证并更新相关文档。
- **使用场景**: 代码逻辑发生了变更，但配套文档（如 README、API 接口文档）已过时，需要同步更新时使用。
- **e.g.**: 修改了后端路由后，运行 `/gsd-docs-update` 自动根据新代码重写 API 接口说明。

## /gsd-explore

- **功能简述**: 苏格拉底式构思与路由
- **详细说明**: 在正式制定计划前进行深度思考和路径选择。
- **使用场景**: 面对模糊的业务需求，不知道从何下手，需要 AI 使用苏格拉底式提问引导你理清思路时使用。
- **e.g.**: `/gsd-explore 我想在系统里加一个会员等级功能，但还没想好具体规则。`

## /gsd-fast

- **功能简述**: 内联执行简单任务
- **详细说明**: 无需子代理或复杂的计划开销，直接快速完成琐碎任务。
- **使用场景**: 遇到非常简单、琐碎的编码或文档任务，不需要繁杂的规划阶段（PLAN.md 等），想要直接内联执行时使用。
- **e.g.**: `/gsd-fast 修复 utils.py 第 45 行的拼写错误。`

## /gsd-forensics

- **功能简述**: 失败工作流的事后调查
- **详细说明**: 诊断 GSD 工作流失败的原因，定位问题根源。
- **使用场景**: GSD 工作流在执行过程中意外崩溃或陷入死循环，需要事后取证并诊断崩溃根源时使用。
- **e.g.**: 自动计划失败后，运行 `/gsd-forensics` 查看是哪一个提示词引发了模型幻觉。

## /gsd-graphify

- **功能简述**: 构建与查询知识图谱
- **详细说明**: 在 .planning/graphs/ 中构建、查询和检查项目知识图。
- **使用场景**: 当项目变得庞大，需要以可视化的知识图谱形式梳理代码依赖关系或业务逻辑流向时使用。
- **e.g.**: 运行 `/gsd-graphify` 生成整个系统的微服务通信依赖图。

## /gsd-health

- **功能简述**: 诊断规划目录健康状况
- **详细说明**: 检查并修复规划目录（planning directory）中的潜在问题。
- **使用场景**: 感觉 AI 生成的规划文档 `.planning/` 目录可能存在状态不一致、文件丢失或格式损坏等情况时使用。
- **e.g.**: 强制中断上一次任务后，运行 `/gsd-health` 自动修复损坏的进度状态文件。

## /gsd-help

- **功能简述**: 显示可用 GSD 命令及使用指南
- **详细说明**: 查看所有可用命令的快速索引和基本用法。
- **使用场景**: 当你不熟悉 GSD 框架的可用命令，或者忘记了某个具体命令的用法，需要快速查阅指令清单时使用。
- **e.g.**: 初次安装 GSD 工具后，输入 `/gsd-help` 查看支持哪些自动化工作流。

## /gsd-import

- **功能简述**: 导入外部计划
- **详细说明**: 在写入前进行冲突检测，确保外部计划与现有项目决策一致。
- **使用场景**: 团队其他成员或外部工具生成了一份计划文档，需要将其安全导入当前项目并进行冲突检查时使用。
- **e.g.**: 运行 `/gsd-import path/to/external_plan.md`，检测它是否与当前项目的架构决策冲突。

## /gsd-inbox

- **功能简述**: 分流并审查 GitHub 问题与 PR
- **详细说明**: 对照项目模板和贡献指南，处理并审查待办事项。
- **使用场景**: 作为项目维护者，需要集中处理 GitHub 仓库中积压的 Issue 或 PR 时使用。
- **e.g.**: 运行 `/gsd-inbox` 自动拉取最新的 PR，并根据贡献指南自动检查代码格式是否合规。

## /gsd-manager

- **功能简述**: 交互式命令控制中心
- **详细说明**: 在一个终端中管理和协调多个阶段的任务。
- **使用场景**: 作为架构师或技术主管，需要同时把控并调度多个并行开发的阶段任务时使用。
- **e.g.**: 运行 `/gsd-manager` 打开交互式面板，分配哪些阶段进入“计划”，哪些进入“执行”。

## /gsd-mvp-phase

- **功能简述**: 规划垂直 MVP 切片阶段
- **详细说明**: 包含用户故事、SPIDR 拆分及阶段计划。
- **使用场景**: 拿到一个大需求后，需要按最小可行性产品 (MVP) 的理念将其拆解为一个可以独立运行的垂直切片时使用。
- **e.g.**: `/gsd-mvp-phase 规划一个只能通过邮箱密码登录的极简认证模块阶段。`

## /gsd-new-project

- **功能简述**: 初始化新项目
- **详细说明**: 进行深度上下文采集并生成初始 PROJECT.md。
- **使用场景**: 从零开始构建一个全新的软件项目，需要搭建初始目录结构、约束规则和初始规划架构时使用。
- **e.g.**: 新建空文件夹后运行 `/gsd-new-project` 引导生成项目说明 (`PROJECT.md`) 及基础设定。

## /gsd-ns-context

- **功能简述**: 代码库情报分析
- **详细说明**: 通过 graphify 文档学习并映射代码库情报。
- **使用场景**: 接手一个新项目或庞大代码库，需要 AI 阅读并理解全局文档与代码结构，建立深层上下文情报时使用。
- **e.g.**: 刚 clone 了一个开源项目，运行 `/gsd-ns-context` 让 AI 分析并映射核心逻辑。

## /gsd-ns-ideate

- **功能简述**: 探索性捕捉
- **详细说明**: 捕捉探索阶段的灵感及草图规格说明。
- **使用场景**: 处理属于 `ns`（命名空间）探索范畴的任务，主要用于创意激发和草案整理时使用。
- **e.g.**: 运行 `/gsd-ns-ideate` 将刚才头脑风暴记录的草图转化为具体的规格说明（Spec）。

## /gsd-ns-manage

- **功能简述**: 配置工作区与线程
- **详细说明**: 更新工作线程、交付物、收件箱及相关配置。
- **使用场景**: 针对特定命名空间配置工作线程，处理多线并发工作流的状态更新与收件箱清理时使用。
- **e.g.**: 运行 `/gsd-ns-manage` 整理当前工作区的待办邮件和任务线程状态。

## /gsd-ns-project

- **功能简述**: 项目生命周期审计
- **详细说明**: 汇总项目生命周期与里程碑审计摘要。
- **使用场景**: 需要对整个项目的生命周期进行回顾，或者生成针对里程碑的总体执行报告时使用。
- **e.g.**: 第一季度开发结束，运行 `/gsd-ns-project` 生成里程碑审计与总结报告。

## /gsd-ns-review

- **功能简述**: 质量门禁与审计
- **详细说明**: 包含代码审查、调试审计、安全性评估及 UI 检查。
- **使用场景**: 在命名空间层面进行统一的质量门禁审计，包含代码、安全、调试及界面的全面检查时使用。
- **e.g.**: 运行 `/gsd-ns-review` 在代码合入 `main` 前触发强制的安全性评估和 UI 检查。

## /gsd-pause-work

- **功能简述**: 暂停中途工作并移交
- **详细说明**: 在阶段中途暂停时，创建上下文移交文档。
- **使用场景**: 临近下班或被紧急 Bug 打断，需要冻结当前阶段的工作进度并生成移交清单时使用。
- **e.g.**: 运行 `/gsd-pause-work`，AI 会记录当前卡在哪一行代码及后续打算怎么做，方便明天无缝接手。

## /gsd-phase

- **功能简述**: 管理 ROADMAP.md 中的阶段
- **详细说明**: 对路线图中的阶段进行增、删、改、查等 CRUD 操作。
- **使用场景**: 需要管理 `ROADMAP.md` 路线图中的任务阶段（增删改查）时使用。
- **e.g.**: `/gsd-phase insert 在阶段 2 和阶段 3 之间插入一个新的安全审计阶段。`

## /gsd-plan-phase

- **功能简述**: 创建详细阶段计划
- **详细说明**: 生成带有验证循环的 PLAN.md 详细方案。
- **使用场景**: 已经明确了要做什么（Spec），需要 AI 开始思考“怎么做”，并输出包含详细执行步骤的方案时使用。
- **e.g.**: `/gsd-plan-phase 为支付模块出具详细的实施计划 PLAN.md。`

## /gsd-pr-branch

- **功能简述**: 创建干净的 PR 分支
- **详细说明**: 过滤掉 .planning/ 提交，准备好可供审查的干净代码。
- **使用场景**: 在本地完成了开发，想要清理提交历史（移除由 AI 规划工具自动生成的冗余提交），以便呈现给人类 Reviewer 时使用。
- **e.g.**: 开发完毕后运行 `/gsd-pr-branch`，生成一个剔除了 `.planning` 文件夹变更记录的纯净分支。

## /gsd-progress

- **功能简述**: 检查进度或推进工作流
- **详细说明**: 统一的 GSD 状态指令，用于查看进展或派发自由意图。
- **使用场景**: 随时想知道当前任务推进到哪一步了，或者需要一个统一入口推进后续工作流时使用。
- **e.g.**: 任务执行到一半，运行 `/gsd-progress` 查看当前还剩哪些文件未修改。

## /gsd-quick

- **功能简述**: 在 GSD 保证下执行快速任务
- **详细说明**: 具备原子提交和状态跟踪等 GSD 核心特性，但跳过可选的代理环节。
- **使用场景**: 需要快速执行任务，但又希望保留原子提交和状态跟踪等 GSD 核心特性（跳过复杂的子代理规划）时使用。
- **e.g.**: `/gsd-quick 将数据库连接池的超时时间从 30s 修改为 60s。`

## /gsd-review

- **功能简述**: 请求跨 AI 对等审查
- **详细说明**: 从外部 AI CLI 获取对项目计划的对等评审意见。
- **使用场景**: 计划制定完成后，希望通过另一个独立的 AI 代理（如使用不同的模型）进行交叉验证和审查时使用。
- **e.g.**: 运行 `/gsd-review` 邀请 Claude 模型对当前生成的 `PLAN.md` 提出安全漏洞审查意见。

## /gsd-settings

- **功能简述**: 配置工作流与模型
- **详细说明**: 调整 GSD 工作流逻辑和底层模型配置。
- **使用场景**: 类似于 `gsd-config`，更侧重于调整 GSD 框架的底层模型选择和工作流具体策略。
- **e.g.**: 运行 `/gsd-settings` 开启自动修复测试代码的功能选项。

## /gsd-ship

- **功能简述**: 创建 PR 并准备合并
- **详细说明**: 运行代码审查，在验证通过后为 PR 的合并做最后准备。
- **使用场景**: 某个功能分支的代码已编写完毕，准备提 PR (Pull Request) 合并到主分支前，需要进行最后审查和准备时使用。
- **e.g.**: 阶段任务完成后，运行 `/gsd-ship` 自动进行代码审查、运行验证并生成 PR 描述草稿。

## /gsd-sketch

- **功能简述**: 草绘 UI/设计创意
- **详细说明**: 使用丢弃式的 HTML 原型勾勒构思，或提出后续的设计方案。
- **使用场景**: 在开发前端功能前，需要快速生成带有交互的 HTML/CSS 原型草图以验证设计想法时使用。
- **e.g.**: `/gsd-sketch 为登录页面生成一个包含深色模式切换的流体设计原型。`

## /gsd-spec-phase

- **功能简述**: 明确交付物并评分
- **详细说明**: 在讨论阶段前产出 SPEC.md，解决需求模糊性。
- **使用场景**: 在讨论阶段开始前，需求还不够清晰，需要强制规范交付物标准并进行需求模糊度评分时使用。
- **e.g.**: `/gsd-spec-phase 将这段含糊的产品说明转化为格式化的 SPEC.md 契约文档。`

## /gsd-spike

- **功能简述**: 技术预研或探索
- **详细说明**: 通过实验性探索来验证想法，或提出下一步的预研建议。
- **使用场景**: 在正式开发某个复杂功能前，需要进行技术预研（Spike）、探索可行性或进行实验性编码时使用。
- **e.g.**: `/gsd-spike 调研并将 Redis 替换为 DragonflyDB 的可行性。`

## /gsd-stats

- **功能简述**: 显示项目统计信息
- **详细说明**: 查看阶段、计划、需求、Git 指标和时间轴等数据。
- **使用场景**: 想要了解当前项目的宏观进度，包括已完成阶段数、剩余任务、代码提交指标等数据面板时。
- **e.g.**: 每日晨会前运行 `/gsd-stats`，查看当前里程碑的进度条和 Git 活跃度。

## /gsd-thread

- **功能简述**: 管理持久化上下文对话
- **详细说明**: 维护跨会话的上下文线程，确保工作连贯性。
- **使用场景**: 需要在不同时间段继续同一个复杂任务，要求持久化保存上下文以保证对话连贯性时使用。
- **e.g.**: `/gsd-thread resume id-1234` 恢复昨天未完成的数据库迁移讨论。

## /gsd-ui-phase

- **功能简述**: 生成 UI 设计契约
- **详细说明**: 为前端阶段生成 UI-SPEC.md 设计规范。
- **使用场景**: 在启动一个纯前端界面的开发阶段前，需要严格定义界面契约和视觉规范时使用。
- **e.g.**: 针对“用户注册阶段”，运行 `/gsd-ui-phase` 生成包含配色方案和组件状态的 `UI-SPEC.md`。

## /gsd-ui-review

- **功能简述**: 前端代码视觉审计
- **详细说明**: 对已实现的前端代码进行回溯性的“六大支柱”视觉检查。
- **使用场景**: 前端页面开发完成后，需要通过 AI 进行美学和响应式的视觉合规检查时使用。
- **e.g.**: 页面写完后运行 `/gsd-ui-review`，检查是否符合“六大支柱”视觉标准（如对比度、对齐等）。

## /gsd-undo

- **功能简述**: 安全 Git 回滚
- **详细说明**: 根据阶段清单（phase manifest）及其依赖关系检查，安全地撤销阶段或计划提交。
- **使用场景**: 发现刚完成的阶段任务方向错误或引发了严重问题，需要安全地撤销相关代码提交并保持依赖完整性时。
- **e.g.**: 运行 `/gsd-undo phase-3`，系统会根据清单安全回滚阶段 3 的所有 Git 提交。

## /gsd-update

- **功能简述**: 更新 GSD 版本
- **详细说明**: 更新至最新版本并显示变更日志（Changelog）。
- **使用场景**: 需要将 GSD CLI 工具升级到最新版本，并查看新功能或修复的 Bug 时使用。
- **e.g.**: 每月定期运行 `/gsd-update` 保持工具链最新并查看 Changelog。

## /gsd-workspace

- **功能简述**: 管理 GSD 工作区
- **详细说明**: 创建、列出或删除隔离的工作环境。
- **使用场景**: 在同一台机器上开发多个独立项目，需要隔离它们的环境变量和 GSD 配置时使用。
- **e.g.**: `/gsd-workspace create new-saas-project` 创建一个干净的开发上下文。

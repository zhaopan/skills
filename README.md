# 技能库工程规约 (Skills Repository Engineering Conventions)

本仓库定义了一系列 AI 代理工程域技能模块，用于标准化软件开发工作流并确立质量控制基准。下表列出了各模块的技术逻辑定义与执行约束：

| 工程域 | 技术逻辑定义 | 执行上下文与约束 |
| :--- | :--- | :--- |
| **adapt-agent-configs** | 自动识别并解析第三方 AI 代理 CLI（Gemini, Kiro, Copilot）配置，执行工作流模式向当前工程环境的逻辑转换与映射。 | 会话初始化、跨平台工作流迁移、代理配置继承、多工具链共存环境。 |
| **api-and-interface-design** | 抽象并定义高稳定性、文档完备且具备强类型约束的交互契约。 | 系统边界定义、跨模块协议对接、组件属性接口规范、数据库模式定义、公共接口变更。 |
| **browser-testing-with-devtools** | 通过 Chrome DevTools MCP 注入浏览器运行时状态，建立静态源码分析与动态渲染环境的逻辑映射。 | 浏览器渲染内容构建、UI 异常诊断、控制台错误溯源、网络请求分析、性能剖析、修复验证。 |
| **ci-cd-and-automation** | 部署自动化质量准入协议，对代码变更实施全量回归验证、合规性审计与构建约束校验。 | CI 流水线初始化、自动化校验规约配置、部署准入逻辑定义、回归验证执行、流水线故障排查。 |
| **code-review-and-quality** | 实施多维代码审计，涵盖算法正确性、架构一致性、安全性评价及性能基准验证。 | 变更合并前置审计、功能实现质量评估、存量代码结构化重构、缺陷修复回归审查。 |
| **code-simplification** | 在等效逻辑约束下实施结构化重构，优化代码圈复杂度与系统维护熵。 | 复杂度超标逻辑处理、设计优化建议响应、深层嵌套逻辑消解、应急代码规范化重构。 |
| **context-engineering** | 基于任务生命周期实施动态上下文拓扑管理，收敛模型幻觉并确立输出一致性标准。 | 会话初始化、模型响应质量震荡处理、任务域切换、项目上下文配置、规范偏离纠正。 |
| **debugging-and-error-recovery** | 实施结构化异常分诊，通过故障重现与根因追踪确保逻辑缺陷的闭环修正。 | 测试失败修复、构建中断处理、运行时异常排查、缺陷报告响应、错误日志溯源分析。 |
| **deprecation-and-migration** | 执行特性生命周期管理协议，实施过时组件的渐进式剥离与版本迁移。 | 陈旧系统/库替换、冗余特性移除、重复实现整合、无维护代码清理。 |
| **documentation-and-adrs** | 维护架构决策记录（ADRs），持久化设计约束、技术选型依据及其演进拓扑。 | 重大架构决策确立、技术选型分析、公共 API 契约变更、破坏性特性发布、新成员入职同步。 |
| **frontend-ui-engineering** | 遵循原子化设计系统规范构建生产环境界面，强制执行可访问性、交互一致性与性能基准约束。 | UI 组件/页面构建、交互界面修补、响应式布局实现、组件状态拓扑管理。 |
| **git-workflow-and-versioning** | 实施标准化版本控制规约，强制执行原子化提交与短周期分支隔离策略。 | 代码版本变更、原子化 Commit 提交、分支策略控制、合并冲突处理、并行开发流调度。 |
| **idea-refine** | 应用结构化解构方法，将非形式化语义提炼为具备逻辑可行性的技术原型。 | 原始需求方案完善、技术概念建模、工程计划压力测试。 |
| **incremental-implementation** | 实施增量构建协议，通过垂直切片拆分确立迭代周期的原子化验证机制。 | 多文件变更执行、基于任务拆分的功能构建、逻辑重构实施、提交规模阈值控制。 |
| **performance-optimization** | 基于基准测试定位执行瓶颈，实施针对性逻辑优化以收敛 SLA 偏离度。 | 性能指标（SLA）响应、系统响应缓慢溯源、核心 Web 指标优化、性能退化排查。 |
| **planning-and-task-breakdown** | 实施任务解耦与粒度细化，定义原子化验收准则与任务依赖有向图。 | 技术规格说明书解构、超大规模任务评估、开发周期规划、并行开发依赖管理。 |
| **security-and-hardening** | 实施安全加固规约，强制执行输入过滤、身份认证及敏感数据加密约束。 | 外部输入处理、身份验证/授权逻辑实现、敏感数据存储加固、外部 API 集成、上传安全管理。 |
| **shipping-and-launch** | 实施发布保障协议，涵盖运行环境预检、渐进式部署及状态回滚逻辑。 | 功能初次发布执行、重大变更部署、数据模式迁移管理、Beta 测试阶段受控发布。 |
| **source-driven-development** | 以框架官方文档作为权威逻辑源，确保实现方案契合框架核心约束与版本特性。 | 框架最佳实践遵循、样板代码构建、框架核心模式实现、代码合规性深度审计。 |
| **spec-driven-development** | 在编码阶段前确立结构化技术规格书，定义系统实现的逻辑基准与边界条件。 | 项目/功能定义初始化、模糊需求形式化、跨模块变更协调、架构设计准则确立。 |
| **test-driven-development** | 实施测试驱动开发协议，通过失败断言驱动业务逻辑的精确实现。 | 业务逻辑实现、已知缺陷修复、存量功能修正、边缘情况约束处理。 |
| **using-agent-skills** | 动态检索并映射任务阶段对应的工程域技能，执行特定工作流逻辑规约。 | 会话起始阶段技能检索、当前任务最优工程路径确立。 |

## 工程规约

依据任务生命周期阶段检索对应 `SKILL.md` 文档，获取工作流逻辑定义与执行规范。所有操作必须符合以下约束：
- **变更域隔离协议**：严格限制代码修改于目标任务域，严禁执行非相关的附属重构。
- **架构极简约束**：优先采用低复杂度、高内聚的逻辑实现方案，严禁过度设计。
- **歧义消解协议**：识别并消除非形式化需求中的不确定性，严禁基于模糊前提执行逻辑推演。
- **原子化断言机制**：所有状态变更必须通过自动化测试或运行时断言进行验证。

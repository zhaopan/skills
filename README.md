# 技能库汇总 (Skills Repository)

本项目包含一系列专业 AI 代理技能，旨在规范工程工作流并提升开发质量。以下是各技能模块的详细汇总：

| 技能名称 | 描述 | 主要用途 |
| :--- | :--- | :--- |
| **API and Interface Design** | 设计稳定、文档完善且难以误用的接口。适用于 REST API、GraphQL 架构、模块边界等。 | 设计新 API 端点、定义模块边界、创建组件属性接口、建立数据库模式、更改现有公共接口。 |
| **Browser Testing with DevTools** | 利用 Chrome DevTools MCP 为代理提供浏览器视角，桥接静态代码分析与实时浏览器执行。 | 构建或修改浏览器渲染内容、调试 UI 问题、诊断控制台错误、分析网络请求、分析性能、验证修复。 |
| **CI/CD and Automation** | 自动化质量门禁，确保变更在进入生产环境前通过测试、代码检查、类型检查和构建。 | 设置新项目的 CI 流水线、添加或修改自动化检查、配置部署流水线、触发自动化验证、调试 CI 失败。 |
| **Code Review and Quality** | 多维度代码审查，涵盖正确性、可读性、架构、安全和性能五个核心维度。 | 合并 PR 或变更前、完成功能实现后、评估他人代码、重构现有代码、修复 Bug 后。 |
| **Code Simplification** | 在保持行为不变的前提下降低代码复杂度，提升代码的可读性、可维护性和可调试性。 | 功能运行正常但实现过重、代码审查发现复杂度问题、逻辑嵌套过深、重构压力下编写的代码。 |
| **Context Engineering** | 在正确的时间为代理提供正确的信息，通过精细化的上下文管理提升输出质量。 | 开始新会话、代理输出质量下降、切换任务区域、设置新项目、代理未遵守规范。 |
| **Debugging and Error Recovery** | 系统化的 Bug 调试与结构化分诊，强调重现、定位、缩小范围、修复根因及防止复发。 | 测试失败、构建中断、运行时行为异常、收到 Bug 报告、日志出现错误。 |
| **Deprecation and Migration** | 管理代码的生命周期，安全地移除旧系统、API 或功能，并引导用户完成迁移。 | 替换旧系统或库、移除不再需要的功能、整合重复实现、移除无人维护的死代码。 |
| **Documentation and ADRs** | 记录决策背后的原因（Why），而非仅仅是代码做什么（What），包括架构决策记录（ADRs）。 | 做出重大架构决策、选择不同方案、更改公共 API、发布改变用户行为的功能、成员入职。 |
| **Frontend UI Engineering** | 构建生产级、可访问且性能优异的用户界面，强调遵循设计系统和避免 AI 生成感。 | 构建新 UI 组件或页面、修改现有界面、实现响应式布局、添加交互或状态管理。 |
| **Git Workflow and Versioning** | 规范 Git 工作流实践，通过原子提交、描述性信息和短生命周期分支保持历史清晰。 | 进行任何代码变更、提交、分支、冲突解决、组织并行工作流。 |
| **Idea Refine** | 通过结构化的发散和收敛思维，将原始想法提炼为具有可操作性的概念。 | 完善原始想法、进行概念构思、对计划进行压力测试。 |
| **Incremental Implementation** | 通过垂直切片的方式增量构建功能，确保每一步都可测试、可验证。 | 实现多文件变更、基于任务分解构建功能、重构代码、避免一次性编写过多代码。 |
| **Performance Optimization** | 基于衡量的性能优化，识别真实瓶颈并针对性修复，避免过早优化。 | 存在性能需求或 SLA、收到运行缓慢的报告、核心 Web 指标不达标、怀疑出现性能退化。 |
| **Planning and Task Breakdown** | 将工作分解为细小的、可验证的任务，并明确验收标准和依赖关系。 | 有规格说明需分解为实现单元、任务过大无法开始、评估范围、并行开发。 |
| **Security and Hardening** | 安全优先的开发实践，防御常见漏洞，确保输入校验、身份验证和机密管理。 | 处理用户输入、实现身份验证或授权、存储敏感数据、集成外部 API、文件上传。 |
| **Shipping and Launch** | 确保生产发布的安全性和可观测性，包括预发布检查清单、灰度发布和回滚策略。 | 首次发布新功能、发布重大变更、迁移数据或架构、进行 Beta 测试。 |
| **Source-Driven Development** | 所有框架相关的决策必须以官方文档为准，确保代码的正确性和时效性。 | 遵循框架最佳实践、构建样板代码、实现涉及框架核心模式的功能、审查代码。 |
| **Spec-Driven Development** | 在编写代码前编写结构化的规格说明书，作为团队协作和实现的唯一真实来源。 | 启动新项目或功能、需求模糊不清、涉及多模块变更、做出架构决策。 |
| **Test-Driven Development** | 在编写实现代码前先编写失败的测试，通过测试驱动逻辑实现和 Bug 修复。 | 实现新逻辑、修复 Bug（证明模式）、修改现有功能、添加边界处理。 |
| **Using Agent Skills** | 发现并调用代理技能的元技能，根据开发阶段选择合适的工程工作流。 | 开始会话时发现适用技能、确定当前任务的最佳工程实践路径。 |

## 使用说明

请根据当前任务所处的开发阶段，参考上述表格选择对应的 `SKILL.md` 文件获取详细的工作流指导。所有技能均遵循以下核心原则：
- **主动管理模糊性**：在遇到不一致或不明确的需求时停止并询问。
- **推崇简单性**：避免过度设计，优先采用简单直接的解决方案。
- **验证而非假设**：所有变更必须经过测试或实际运行验证。
- **遵守范围限制**：仅修改任务相关的代码，避免无谓的“顺手清理”。
